# CLAUDE.md

Guidance for AI assistants (and humans) working in this repository.

## What this is

**StreamMulticast** is a native C++ plugin for **OBS Studio** that streams a single OBS
session to N RTMP endpoints (Twitch, YouTube, Facebook, Kick, Trovo, custom) simultaneously,
with **independent encoder backend, bitrate, audio bitrate, keyframe interval, and frame
orientation per output**. It is the free/open-source alternative to the paid features of
Aitum Multistream.

- **Language / standard:** C++17 (a small amount of C in the generated `plugin-support`)
- **Frameworks:** `libobs` + `obs-frontend-api` (OBS plugin API) and **Qt 6** (Widgets/Core/Gui) for the dock UI
- **License:** GPL-2.0-or-later (mandated by libobs being GPL-2.0)
- **Target OBS version:** 31.1.1 (see `buildspec.json`); aims to stay compatible with OBS 30.x–31.x
- **Shipping platform:** Windows x64 only. macOS/Linux CMake presets exist and are kept
  building, but **only Windows is built in CI and released**. macOS/Linux are roadmap items.

The single artifact is `streammulticast.dll` (~230 KB), installed into an OBS install
alongside `data/obs-plugins/streammulticast/locale/`.

## Architecture

Single-source / multi-encoder design: the OBS main video mix is tapped once, then N parallel
`obs_encoder_t` instances feed N parallel `obs_output_t` (rtmp_output) outputs. Code is split
into three layers under `src/`, all in namespace **`smulti`**.

```
src/
├── plugin-main.cpp        Module entry. obs_module_load/unload, owns the module-level
│                          singletons, frontend-event handler (linked-to-main start/stop),
│                          registers the dock via obs_frontend_add_dock_by_id.
├── plugin-support.c.in    Generated by CMake (configure_file) → plugin-support.c.
│   plugin-support.h       Provides obs_log() and PLUGIN_VERSION. Do NOT edit the generated .c.
│
├── core/                  Data + persistence. No Qt, no streaming.
│   ├── Endpoint           Per-output config POD (trivially copyable, passed across
│   │                      threads by value). UUID = stable identity. serialize/
│   │                      deserialize round-trips through obs_data_t.
│   ├── ConfigStore        JSON persistence to the OBS plugin_config dir. 500 ms
│   │                      debounced background-thread save; synchronous save() on unload.
│   ├── EndpointRegistry   Runtime source of truth. Owns one OutputController per
│   │                      endpoint. Observer pattern (UI subscribes for Added/Removed/
│   │                      Updated). Mutex-guarded; observers fired on the Qt UI thread.
│   └── ObsServiceImport   v1.0.6 "Import from OBS": reads the active OBS profile's
│                          service.json / global.ini to pull server URL + stream key.
│
├── pipeline/              The streaming engine. libobs, no Qt.
│   ├── EncoderFactory     Stateless. Creates video (x264/NVENC/QSV/AMF) + AAC audio
│   │                      encoders; reports which backends this machine actually has.
│   ├── OutputController   1 per endpoint. State machine
│   │                      (Idle→Starting→Live→Reconnecting→FailedHard) + reconnect
│   │                      backoff thread (1/2/5/10/30/60 s, max 10 attempts).
│   └── HealthSampler      Background thread polling all outputs at 2 Hz; computes a
│                          rolling 5 s bitrate average and stores thread-safe snapshots.
│
└── ui/                    Qt. Knows about core + pipeline, never the reverse.
    ├── MultistreamDock    QWidget dock with a QTabWidget (Health + Configure).
    ├── HealthTab          Live QTableView grid driven by HealthSampler snapshots.
    ├── ConfigTab          Endpoint cards, add/remove/reorder.
    └── EndpointDialog     Modal editor (encoder, bitrate, orientation, Import from OBS).
```

**Dependency direction:** `ui → pipeline → core`. Core never depends on Qt or on pipeline.
Keep it that way — it is what lets the core data/serialization logic stay testable and
trivially copyable across threads.

### Runtime lifecycle
`obs_module_load()` boots the singletons in order: `ConfigStore` (load from disk) →
`EndpointRegistry` (populate from store) → `HealthSampler` (start polling thread) →
frontend event callback → `MultistreamDock` (handed to Qt, which then owns it).
`obs_module_unload()` tears down in reverse: stop outputs, stop sampler, synchronous
config save, reset registry/store. **Do not manually delete `g_dock`** — Qt owns it after
`obs_frontend_add_dock_by_id`.

### Threading model
Four kinds of threads touch shared state — be careful when changing any of them:
- **Qt UI thread** — all widget code; calls `EndpointRegistry::add/update/remove`.
- **Per-output reconnect thread** (`OutputController`) — guards `state_`/`last_error_` with a mutex.
- **HealthSampler thread** — 2 Hz; reads `obs_output_*` stat getters (assumed thread-safe) and writes snapshots under a mutex.
- **ConfigStore save thread** — debounced disk writes.

Cross-thread → UI updates **must** go through Qt queued connections
(`QMetaObject::invokeMethod`, `Qt::QueuedConnection`). `HealthSampler` and `EndpointRegistry`
only store/emit data; they never touch Qt objects directly.

## Conventions

### `AVANATRO-VERIFY` markers — important
There are ~40 comments tagged **`AVANATRO-VERIFY`** across the source. Each one flags an
assumption about a specific `libobs` / `obs-frontend-api` function (exact signature, lifetime/
ownership rules, thread-safety, or which OBS version added it) that should be confirmed against
the real OBS 31.x headers. They cluster around encoder lifetime vs. output stop ordering,
`obs_data_*` ownership, `bfree()` of returned strings, and dock/frontend API signatures.

When working here:
- **Do not delete an `AVANATRO-VERIFY` marker** unless you have actually verified the claim
  against OBS headers/source and the code is correct — then replace it with a short note of
  what you confirmed.
- When adding new libobs API calls whose contract you can't confirm, **add a new
  `AVANATRO-VERIFY` comment** in the same style rather than silently assuming.

### Code style
- Tabs for indentation, OBS/`obs-plugintemplate` house style (matches the surrounding files).
- Namespace everything in `smulti`. libobs C types are forward-declared in headers
  (`struct obs_output; typedef struct obs_output obs_output_t;`) to avoid pulling `obs.h`
  into every translation unit — keep doing this.
- GPL header block at the top of every source file (see existing files for the two variants).
- Doc comments (`/** ... */`) on public types/methods describe ownership and threading. Match
  that density.
- Formatting tooling lives in `build-aux/` (`run-clang-format`, `run-gersemi` for CMake).
  Run these before committing if you touched C++ or CMake.

### Strings / localization
User-facing strings go in `data/locale/en-US.ini` and are fetched via `obs_module_text("Key")`.
Add a key there rather than hardcoding UI text.

## Build & CI

Versions and dependency hashes are pinned in `buildspec.json` (OBS sources, prebuilt obs-deps,
Qt6). The CMake configure step downloads these into `.deps/` (gitignored) on first run.

```powershell
# Windows (the only shipped platform)
cmake --preset windows-x64                                   # downloads deps on first run
cmake --build --preset windows-x64 --config RelWithDebInfo
cmake --install build_x64 --config RelWithDebInfo --prefix "C:\Program Files\obs-studio"  # admin
```

Available presets (`CMakePresets.json`): `windows-x64`, `windows-ci-x64`, `macos`, `macos-ci`,
`ubuntu-x86_64`, `ubuntu-ci-x86_64`. CI presets add `CMAKE_COMPILE_WARNING_AS_ERROR=ON`, so
**warnings fail the CI build** — keep it clean.

**CI** (`.github/workflows/build-windows.yaml`): builds Windows x64 `RelWithDebInfo` on push to
`main`/`develop`, on PRs to `main`, and on `v*` tags. Pushing a `v*` tag additionally publishes
a **draft** GitHub Release with the zipped artifact. There is no automated test job — validation
is the compile (warnings-as-errors) plus manual testing inside OBS.

### Versioning
`PLUGIN_VERSION` is derived from the `version` field in `buildspec.json` (parsed in
`cmake/common/bootstrap.cmake`) — that file is the single source of truth for the version.
> Note: `buildspec.json` currently reads `1.0.0` while the latest release/tag is `v1.0.6`.
> When cutting a release, bump `buildspec.json` to match the tag.

## Runtime data & privacy
- Config (including **stream keys in plaintext**) is persisted to
  `%APPDATA%\obs-studio\plugin_config\streammulticast\config.json` via
  `obs_module_get_config_path()`. This mirrors how OBS stores its own stream key.
- The plugin makes **no** network connections other than to the user-configured RTMP
  endpoints — no telemetry, no cloud, no accounts. Keep it that way; do not add outbound
  calls without an explicit, documented reason.

## Feature status (see README for detail)
- **v1.0** — multi-RTMP, per-output encoder/bitrate, live Health grid, auto-reconnect backoff,
  linked-to-main toggle, endpoint templates.
- **v1.0.5** — per-output orientation: `Vertical1080x1920Letterbox` (parallel horizontal +
  vertical streaming).
- **v1.0.6** — "Import from OBS" (read stream key from active OBS profile).
- **Not yet implemented:** `Vertical1080x1920CenterCrop` (the enum exists but needs a per-output
  `obs_view_t` + custom render path — reserved for v1.1), per-output resolution/FPS, failover,
  OAuth stream-info push, macOS/Linux release builds.

## Working agreements
- This is a GPL-2.0 project linking GPL libobs — keep new files GPL-headered.
- Don't break the `ui → pipeline → core` dependency direction.
- Preserve thread-safety invariants documented in the headers; if you change a threading
  contract, update the doc comment that describes it.
- Treat `AVANATRO-VERIFY` markers as a checklist, not noise.
